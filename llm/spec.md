# Project Specification: Scalable VLM Data Pipeline & Feature Store

## 1. System Architecture & Philosophy

This project is a scalable, memory-efficient data pipeline for training Vision Language Models (VLMs) locally. It ingests multimodal datasets from Hugging Face, decouples heavy image storage from lightweight text metadata, runs a modular text-preprocessing pipeline, and uses an LMDB-backed feature store for CCG-to-Tensor-Network compilations.

The primary goal is to allow fast iteration over many training and evaluation configurations by cleanly separating infrastructure from research logic.

**Core Philosophies:**

1. **Domain-Driven Encapsulation:** Each dataset is a self-contained "Atlas" containing raw images, raw state, and all derived text variations.
2. **Raw Schema Flexibility:** The ingestion layer enforces no schema on text columns. Preprocessing pipelines standardize wildly different dataset formats into a unified structure.
3. **Zero Image Duplication:** Images are written to disk exactly once. All derivations use absolute pointers.
4. **Streaming & Incremental Execution:** Data is processed in chunks using `polars`. Both ingestion and preprocessing are idempotent and support incremental runs.
5. **Idempotency & Resumability:** Every step maintains state. Interruptions resume safely.
6. **In-Memory Consumption:** The Torch Dataset loads all non-image data into RAM at init. No database is queried at training time. Images are loaded per `__getitem__`.
7. **Composition Over Configuration:** Research logic (sampling, pairing, negative selection) is isolated behind protocol boundaries so it can be swapped without touching infrastructure.

## 2. Data Flow

```
Internet
   ↓
Atlas  (ingestion, image download → data_manifest.parquet)
   ↓
Processing Pipeline  (flatten, normalize, CCG compile → populates LMDB)
   ↓
derived_vX/chunk_*.parquet  (sample_id, local_image_path, processed_text, text_hash)
   ↓
Dataset Creator  (enriches atoms from LMDB, applies CompositionStrategy, splits train/val/test)
   ↓
data/datasets/<name>_{train,val,test}.parquet  (task-schema, diagram + symbols embedded)
   ↓
Torch Dataset  (loads parquet into RAM at init, reads images per __getitem__)
```

## 4. Data Schemas (Strict Contracts)

### A. Raw Manifest (`data_manifest.parquet`)

Generated by the Atlas ingestion layer. One row per source sample (e.g. one row per image). Schema is flexible.

- `sample_id` (String): Unique identifier, e.g. `coco_42`. This is the **group key** — all atoms derived from this sample share it.
- `local_image_path` (String): Absolute path to the downloaded image.
- `*` (Any): Arbitrary text columns from the source (e.g. `sentences_raw`, `caption`).

### B. Derived Manifest (`derived_vX/chunk_*.parquet`)

Generated by the Processing Pipeline. **Flattened** — one row per (image, text) atom. Multiple rows may share the same `sample_id`.

- `sample_id` (String): Group key, carried over from the Raw Manifest. Used for train/val/test splitting and reconstruction.
- `local_image_path` (String): Absolute path, duplicated across all atoms in a group.
- `processed_text` (String): Cleaned and standardized text for this atom.
- `text_hash` (String): SHA-256 of `processed_text`. Primary key for LMDB.

**Unique atom reference:** `(sample_id, text_hash)` — the composite key that uniquely identifies one atom.

### C. Feature Store (`data/sentence_mapping/` — LMDB)

Written by the Processing Pipeline. Read by the Dataset Creator.

- **Key:** `text_hash` (bytes)
- **Value:** JSON bytes containing:
  - `"diagram"`: Einsum string for the compiled CCG tensor diagram.
  - `"symbols"`: Serialized tensor network symbols.

### D. Created Dataset (`data/datasets/<name>_{split}.parquet`)

Generated by the Dataset Creator. Schema is **task-specific** — defined by the `CompositionStrategy`. Contains everything the Torch Dataset needs except images.

Minimum fields (always present):
- `local_image_path` (String or List[String]): Absolute path(s) to image(s).
- `diagram` (String): Einsum string, embedded from LMDB.
- `symbols` (String/Bytes): Serialized symbols, embedded from LMDB.

Additional fields depend on the task (e.g. a second image path, a contrastive caption pair, a label).

## 5. Component Requirements

### Component 1: Atlas Layer (`qnlp/core/data_engine/atlas/`)

- `hf_utils.py`: Lazy HF fetch via `pl.scan_parquet`, sliced by cursor. Multi-threaded image download via `ThreadPoolExecutor`. Drops binary columns, returns lightweight DataFrame.
- `atlas.py`: `Atlas` class. Manages state via `metadata.json`. Generates `sample_id = <atlas_name>_<index>`. Appends to `data_manifest.parquet` incrementally with soft atomicity.

### Component 2: Processing Pipeline

**Common steps** (`qnlp/core/data_engine/processing/`):
- `LemmatizeStep`: Normalizes text into grammatically finite sentences (spaCy). Ensures CCG produces Rank-1 tensors.
- `ConformRankStep`: Fixes einsum output rank to 1D.
- `CCGCompilerStep`: Terminal step — hashes text, delta-checks LMDB, compiles unseen texts in a multiprocessing pool, writes `{text_hash → compiled_bytes}` to LMDB.

**Experiment-specific steps** (`qnlp/preprocessing_pipelines/<experiment>/steps.py`):
- `FlatteningStep`: Explodes multi-value columns into one row per atom. Preserves `sample_id` as the group key across all exploded rows.
- `SchemaMappingStep`: Renames source columns to `processed_text`.
- Any other dataset-specific transformations.

**Orchestrator** (`qnlp/core/data_engine/processing/pipeline.py`):
- Anti-join on `sample_id` between `data_manifest.parquet` and `derived_vX/` to find unprocessed groups.
- Processes only the delta in chunks. Appends to `derived_vX/chunk_*.parquet`.
- LMDB population is a side-effect of `CCGCompilerStep`.

### Component 3: Dataset Creator (`qnlp/core/data_engine/dataset_creator/`)

The infrastructure layer for dataset assembly. Orchestrates enrichment, composition, and splitting. Research logic is injected via `CompositionStrategy`.

**`CompositionStrategy` protocol:**

```python
class CompositionStrategy(Protocol):
    def compose(self, atoms: pl.DataFrame) -> pl.DataFrame:
        """
        Takes a DataFrame of enriched atoms (sample_id, local_image_path,
        processed_text, text_hash, diagram, symbols) and returns a DataFrame
        of composite samples in the task-specific schema.
        """
```

Two modes of composition:
- **Reconstruction**: Group atoms by `sample_id`, select and reshape into the target schema. Used when the target format matches the source dataset's natural structure (e.g. COCO → COCO format, Winoground → Wino format).
- **Synthesis**: Sample atoms across `sample_id` groups to construct artificial composite samples (e.g. COCO → Wino format with randomly sampled negatives, or in future with hard negatives).

Reusable strategies live in `qnlp/core/data_engine/dataset_creator/strategies/`.
Experiment-specific wiring lives in `qnlp/preprocessing_pipelines/<experiment>/dataset_creator.py`.

**Infrastructure responsibilities:**
1. Read and concatenate all `derived_vX/chunk_*.parquet` files.
2. Enrich atoms: query LMDB by `text_hash`, attach `diagram` and `symbols` columns.
3. Split atom groups into train/val/test using `sample_id` as the group key (all atoms sharing a `sample_id` land in the same split).
4. Apply `CompositionStrategy` within each split independently to prevent data leakage.
5. Write enriched, composed `<name>_{train,val,test}.parquet` to `data/datasets/`.

### Component 4: Torch Dataset (`qnlp/domain/datasets/`)

The dataloader is fully generic. Since `CompositionStrategy` already encodes the task schema into the parquet, the dataloader does not need to know the task — it reads whatever columns are present and returns them. No subclassing per task.

**`__init__`**:
- Accept `parquet_path` and `image_columns: list[str]` (columns to load as image tensors, default `["local_image_path"]`).
- Load the parquet fully into RAM as a Polars DataFrame. No LMDB connection.

**`__getitem__(idx)`**:
- Fetch row at `idx`.
- For each column in `image_columns`: load from path via `torchvision.io.read_image`, replace the path string with an image tensor.
- Deserialize `diagram` and `symbols` from their stored format.
- Return a dict mapping column names to values. The dict schema is entirely determined by the parquet, which is determined by the `CompositionStrategy`.

**`__len__`**: Number of rows.

The task schema lives in the parquet. Changing task format means changing the `CompositionStrategy`, not the dataloader.

## 6. Technical Constraints

1. **Strictly Polars:** No Pandas. All tabular operations use `polars`.
2. **Absolute Pathing:** `pathlib.Path` everywhere. All paths in Parquet are resolved to absolute via `.resolve()`.
3. **LMDB is write-time only:** Written during processing, read during dataset creation. Never accessed at Torch Dataset runtime.
4. **Composition happens within splits:** `CompositionStrategy.compose()` is called per split after the train/val/test group split. This prevents data leakage in synthesis scenarios.
5. **No new code in legacy directories:** `qnlp/discoviz/`, `qnlp/image_tower/`, `qnlp/utils/` are kept for comparison only.

## 7. Known Limitations & Future Work

- **Multi-image flattening (Winoground):** The current `FlatteningStep` uses `df.explode()` which varies text across rows but holds the image path fixed. Winoground samples (2 images × 2 captions) require a different flattening strategy where the image path also varies. This is a future problem — the dataset creator and strategy layer are designed to accommodate it without infrastructure changes.
- **Hard negative sampling:** `CompositionStrategy` implementations currently use random negative selection. Replacing this with embedding-based or model-in-the-loop hard negative sampling requires only a new `CompositionStrategy` implementation, no infrastructure changes.
